# Repository Guidelines

Concise guide for contributing to the AI Flow CLI and its branch/step workflow.

## Project Structure & Module Organization
- `ai_flow.py`: single-entry CLI with templates, argument parsing, file operations, and diagram builder.
- `project.md`, `plan.md`, `journal.md`: top-level project context files generated by the CLI.
- `branches/<branch_id>/branch-info.md` and `branches/<branch_id>/runs/<step_id>/` (prompt.md, context.md, result_raw.md, evaluation.md) hold branch metadata and per-step artifacts.
- `tests/test_ai_flow.py`: unittest suite covering CLI flows, git interactions, and diagram rendering.
- `README.md`: usage examples and conventions for branches/steps.

## Build, Test, and Development Commands
- `python ai_flow.py --help` to inspect available subcommands.
- `python ai_flow.py init-project <path> --title "Demo"` scaffolds a project, initializes git if missing, and seeds branch `A/001`.
- `python ai_flow.py create-branch <path> <branch_id> [--parent <id>] [--from-step <A_002>] [--status closed]` makes a branch plus its first step.
- `python ai_flow.py new-step <path> <branch_id> <step_id> [--from-step <A_001>]` appends a run; requires a clean git tree when a repo exists.
- `python ai_flow.py diagram <path> --output branches/diagram.mmd` emits a Mermaid diagram of branches/steps.
- Tests: `python -m unittest tests/test_ai_flow.py` (git must be on PATH for git-dependent cases; set `git config user.name/email` in temp repos when needed).

## Coding Style & Naming Conventions
- Python 3 with 4-space indents; favor standard library patterns used in `ai_flow.py`.
- Functions/variables use snake_case; constants are UPPER_SNAKE; keep helpers small and side-effect aware.
- Branch IDs like `A_main`, `B_alt-from-A_002`; step IDs like `001` or `A_001`; templates assume UTF-8 content.
- Preserve deterministic file layouts and avoid hidden state outside the target project path.

## Testing Guidelines
- Add unittest cases under `tests/` named `test_*.py`; mirror new CLI behaviors with success and failure expectations.
- When changing templates or metadata, assert generated file contents and structure (e.g., prompt text, evaluation status lines).
- For git-related flows, ensure repositories are clean before invoking `new-step`; prefer temporary directories to avoid polluting the workspace.

## Commit & Pull Request Guidelines
- No commit history yet; use short imperative messages (`add diagram output`) or conventional prefixes (`feat: diagram export`).
- PRs should summarize the change, list commands run (e.g., `python -m unittest tests/test_ai_flow.py`), link issues/branches, and attach generated diagrams when relevant.
- Keep the working tree clean before running CLI commands that enforce git cleanliness; document any manual follow-up steps in the PR description.

## Security & Configuration Tips
- CLI writes only under the supplied project path; avoid targeting directories with sensitive or untracked data.
- Diagram output is plain Mermaid text; review and scrub confidential names before sharing externally.
