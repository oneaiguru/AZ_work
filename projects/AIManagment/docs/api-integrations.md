# AIManagment — направления для API и интеграций

## Цель документа

Определить минимальный API-контур и зафиксировать, какие интеграции стоит
закладывать в архитектуру сразу, даже если они будут включаться поэтапно.

## Принципы backend-этапа

- backend должен повторять ключевые сценарии текущего MVP без ухудшения UX;
- API должен быть удобен для постепенного перевода фронтенда с `localStorage`;
- интеграции не должны ломать базовый ручной сценарий работы;
- все внешние синхронизации должны быть наблюдаемыми и обратимыми.

## Предлагаемый API v1

Базовый префикс: `/api/v1`

### 1. Workspaces

#### `GET /workspaces/:workspaceId`

Возвращает метаданные workspace и базовые настройки доски.

#### `GET /workspaces/:workspaceId/members`

Возвращает список участников и их роли.

### 2. Tasks

#### `GET /tasks`

Поддерживаемые query-параметры:

- `workspaceId`
- `status`
- `priority`
- `ownerId`
- `modelId`
- `query`
- `limit`
- `cursor`

Назначение:
- загрузка канбан-доски;
- фильтрация;
- серверный поиск;
- пагинация для больших команд.

#### `POST /tasks`

Создаёт задачу.

Пример запроса:

```json
{
  "workspaceId": "ws_prod_ai",
  "title": "Проверить AI triage для support",
  "description": "Сравнить качество маршрутизации тикетов.",
  "status": "backlog",
  "priority": "high",
  "ownerId": "usr_ira",
  "modelId": "mdl_gpt_4_1"
}
```

#### `PATCH /tasks/:taskId`

Изменяет редактируемые поля карточки.

#### `POST /tasks/:taskId/advance`

Сдвигает задачу на следующий статус по workflow. Нужен для сохранения текущего
сценария quick action из MVP.

#### `POST /tasks/:taskId/cycle-priority`

Меняет приоритет по кругу. Это компромиссный endpoint для быстрого действия из
существующего UI; позднее его можно заменить обычным `PATCH /tasks/:taskId`.

#### `POST /tasks/:taskId/archive`

Мягко архивирует задачу без физического удаления.

### 3. Comments and history

#### `GET /tasks/:taskId/comments`
#### `POST /tasks/:taskId/comments`
#### `GET /tasks/:taskId/events`

Эти методы не нужны для первой миграции MVP, но должны быть запланированы в
архитектуре заранее, потому что без них сложно масштабировать командную работу.

### 4. Reference data

#### `GET /models`

Возвращает список доступных AI-моделей для выбора в интерфейсе.

#### `GET /users`

Возвращает участников workspace для назначения владельца задачи.

### 5. Dashboard analytics

#### `GET /dashboard/summary?workspaceId=ws_prod_ai`

Возвращает агрегаты, которые сейчас считаются на клиенте:

```json
{
  "total": 18,
  "completed": 6,
  "inFlight": 7,
  "critical": 3,
  "topModel": "GPT-4.1"
}
```

## Контракты ответа

### Формат успешного ответа

```json
{
  "data": {},
  "meta": {
    "requestId": "req_123"
  }
}
```

### Формат ошибки

```json
{
  "error": {
    "code": "validation_error",
    "message": "status is invalid",
    "details": {
      "field": "status"
    }
  }
}
```

## Интеграции: приоритеты

### Первая волна

#### 1. Jira

Сценарии:
- связывать AI-задачу с продуктовой задачей или багом;
- импортировать ключевые ссылки и внешний идентификатор;
- синхронизировать статус вручную или по кнопке.

Почему в первой волне:
- чаще всего AI-инициативы живут рядом с обычным delivery-процессом;
- команде нужен мост между экспериментами и обычным backlog.

#### 2. Slack

Сценарии:
- отправлять уведомление о переходе критичной задачи в `review` или `done`;
- быстро открывать Slack-thread из карточки задачи;
- принимать slash-команду для создания задачи.

Почему в первой волне:
- Slack снижает риск пропустить изменение статуса;
- интеграция не требует глубокого пересмотра модели данных.

### Вторая волна

#### 3. Notion

Сценарии:
- хранить расширенную документацию или research notes;
- прикреплять ссылку на спецификацию эксперимента;
- использовать как read-only источник контекста.

#### 4. BI / usage providers

Сценарии:
- подтягивать usage или cost-метрики по моделям;
- строить отчёты по эффективности AI-инициатив;
- сравнивать объём экспериментов по командам и направлениям.

## Стратегия синхронизации

### Вариант 1. Link-only

Самый безопасный старт:
- задача хранится в AIManagment как primary record;
- внешняя система даёт только ссылку и идентификатор;
- конфликтов почти нет.

### Вариант 2. One-way sync

Подходит для Jira или Slack событий:
- изменения приходят извне и записываются в `task_events`;
- UI показывает источник обновления;
- бизнес-правила остаются на стороне AIManagment.

### Вариант 3. Two-way sync

Нужно отложить до зрелой версии, потому что появляются:
- конфликты статусов;
- гонки обновлений;
- требования к idempotency и retry-логике.

## Нефункциональные требования к API

- **Аудит:** каждое критичное изменение создаёт `task_event`.
- **Идемпотентность:** интеграционные операции должны уметь безопасно
  повторяться.
- **Наблюдаемость:** у фоновых синков нужны статусы, время последней попытки и
  текст ошибки.
- **Версионирование:** публичный API должен жить под `/api/v1`.
- **Безопасность:** workspace-scoped доступ и ролевая модель без глобальных
  администраторских исключений на уровне приложения.

## Рекомендуемая последовательность реализации

1. Поднять CRUD для задач и справочники пользователей/моделей.
2. Подключить endpoint сводки дашборда.
3. Перевести фронтенд на чтение с API при сохранении локального fallback режима.
4. Добавить `task_events` и comments.
5. Подключить link-only интеграции с Jira и Slack.
6. Только потом обсуждать двустороннюю синхронизацию.

## Что можно не делать в первой backend-итерации

- real-time обновления через WebSocket;
- drag-and-drop с мгновенной синхронизацией;
- полную RBAC-матрицу на уровне отдельных полей;
- сложные automation rules;
- двусторонний sync с несколькими внешними системами одновременно.
