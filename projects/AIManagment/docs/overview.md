# AIManagment — обзор приложения

## Идея

AIManagment — это лёгкое веб-приложение для управления AI-задачами внутри команды.
Оно помогает фиксировать инициативы, отслеживать статус экспериментов,
контролировать используемые модели и видеть общую загрузку по направлениям.

## Проблема

Когда команда одновременно тестирует несколько AI-сценариев, быстро появляется хаос:

- задачи расползаются по чатам и заметкам;
- непонятно, какие модели уже пробовали;
- сложно увидеть, что находится в работе, а что уже готово;
- статус инициатив зависит от памяти участников, а не от единого источника данных.

## Цель MVP

Собрать в одном интерфейсе минимально достаточный набор возможностей:

1. добавить AI-задачу;
2. указать владельца, модель, приоритет и статус;
3. фильтровать задачи по статусу и поисковому запросу;
4. видеть ключевые метрики по текущему набору задач;
5. хранить состояние локально, без сервера.

## Сценарии использования

### 1. Руководитель AI-направления

- смотрит сводку по количеству задач;
- быстро понимает, сколько задач в работе и сколько завершено;
- видит высокоприоритетные инициативы.

### 2. Инженер или аналитик

- добавляет новую инициативу;
- указывает модель (`GPT-4.1`, `GPT-4o`, `Claude`, `Gemini` и т.д.);
- переводит задачу по статусам по мере работы.

### 3. Команда продукта

- использует приложение как визуальную доску;
- просматривает backlog, работу, ревью и завершённые задачи;
- фильтрует карточки по ключевому слову.

## Границы первой версии

В первую версию **не** входят:

- авторизация;
- сервер и база данных;
- совместная работа в реальном времени;
- drag-and-drop;
- интеграции с Jira / Notion / Slack;
- экспорт в файлы.

## Технический подход

Для первого шага достаточно статического фронтенда:

- `index.html` — структура интерфейса;
- `styles.css` — базовая адаптивная стилизация;
- `app.js` — рендер, события и работа с DOM;
- `state.js` — чистые функции для задач, фильтров и статистики;
- `localStorage` — персистентность между перезагрузками.

Такой подход позволит быстро проверить UX и структуру данных, а уже затем при
необходимости вынести состояние на backend. В текущем прототипе отдельное
внимание уделяется мобильному сценарию: фильтры и быстрые статус-вкладки
должны позволять просматривать доску с телефона без потери контекста.
