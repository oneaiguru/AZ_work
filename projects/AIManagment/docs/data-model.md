# AIManagment — модель данных backend-этапа

## Цель документа

Зафиксировать переход от локального `localStorage`-состояния к серверной модели,
которая поддержит командную работу, историю изменений и будущие интеграции.

## От чего отталкиваемся

Сейчас фронтенд хранит плоский массив задач со следующими полями:

- `id`
- `title`
- `owner`
- `model`
- `priority`
- `status`
- `description`
- `createdAt`

Для backend-версии этого достаточно как стартового контракта, но не хватает:

- нормализованных сущностей пользователей и моделей;
- истории изменений статуса и приоритета;
- системных временных меток обновления;
- связей с внешними системами;
- командного контекста и ролей доступа.

## Основные сущности

### 1. Workspace

Контейнер команды или направления, в рамках которого ведутся AI-задачи.

```json
{
  "id": "ws_prod_ai",
  "name": "Product AI",
  "slug": "product-ai",
  "createdAt": "2026-03-19T12:00:00.000Z",
  "updatedAt": "2026-03-19T12:00:00.000Z"
}
```

### 2. User

Участник команды, который может быть владельцем задачи, автором изменений или
подписчиком уведомлений.

```json
{
  "id": "usr_olga",
  "workspaceId": "ws_prod_ai",
  "name": "Olga Petrova",
  "email": "olga@example.com",
  "role": "manager",
  "isActive": true,
  "createdAt": "2026-03-19T12:00:00.000Z",
  "updatedAt": "2026-03-19T12:00:00.000Z"
}
```

### 3. ModelCatalogItem

Каталог доступных моделей, чтобы уйти от свободного текстового поля и получить
аналитику по использованию провайдеров.

```json
{
  "id": "mdl_gpt_4_1",
  "provider": "OpenAI",
  "name": "GPT-4.1",
  "family": "gpt-4.1",
  "status": "active",
  "contextWindow": 128000,
  "createdAt": "2026-03-19T12:00:00.000Z",
  "updatedAt": "2026-03-19T12:00:00.000Z"
}
```

### 4. Task

Главная сущность системы. Именно она должна остаться максимально близкой к
текущему MVP, чтобы миграция прошла безболезненно.

```json
{
  "id": "tsk_001",
  "workspaceId": "ws_prod_ai",
  "title": "Подготовить AI FAQ для поддержки",
  "description": "Собрать вопросы клиентов и подготовить черновик базы ответов.",
  "status": "in_progress",
  "priority": "high",
  "ownerId": "usr_olga",
  "reporterId": "usr_anna",
  "modelId": "mdl_gpt_4_1",
  "sourceType": "manual",
  "dueDate": null,
  "createdAt": "2026-03-19T08:00:00.000Z",
  "updatedAt": "2026-03-19T13:10:00.000Z",
  "archivedAt": null
}
```

### 5. TaskComment

Нужна для асинхронного взаимодействия команды без вынесения обсуждений в чаты.

```json
{
  "id": "cmt_001",
  "taskId": "tsk_001",
  "authorId": "usr_max",
  "body": "Нужно добавить примеры для e-commerce кейсов.",
  "createdAt": "2026-03-19T13:20:00.000Z"
}
```

### 6. TaskEvent

Единый журнал событий по задаче: смена статуса, приоритета, владельца,
редактирование описания, синхронизация с интеграцией.

```json
{
  "id": "evt_001",
  "taskId": "tsk_001",
  "actorId": "usr_olga",
  "type": "status_changed",
  "payload": {
    "from": "backlog",
    "to": "in_progress"
  },
  "createdAt": "2026-03-19T13:15:00.000Z"
}
```

### 7. ExternalLink

Связь задачи с внешней системой, чтобы одна AI-задача могла быть привязана к
карточке в Jira, странице в Notion или треду в Slack.

```json
{
  "id": "ext_001",
  "taskId": "tsk_001",
  "provider": "jira",
  "externalId": "AIP-42",
  "url": "https://jira.example.com/browse/AIP-42",
  "syncStatus": "connected",
  "lastSyncedAt": "2026-03-19T13:30:00.000Z"
}
```

## Справочники и перечисления

### Task status

- `backlog`
- `in_progress`
- `review`
- `done`
- `blocked` — добавить на backend-этапе для реального сценария работы команды.
- `archived` — служебный статус для скрытия устаревших задач.

### Task priority

- `low`
- `medium`
- `high`
- `critical`

### User role

- `admin`
- `manager`
- `member`
- `viewer`

### Source type

- `manual`
- `jira`
- `notion`
- `slack`
- `api`

## Рекомендуемая структура хранения

### Таблицы / коллекции

1. `workspaces`
2. `users`
3. `model_catalog`
4. `tasks`
5. `task_comments`
6. `task_events`
7. `external_links`

### Базовые связи

- `workspace 1 -> N users`
- `workspace 1 -> N tasks`
- `user 1 -> N owned tasks`
- `model_catalog_item 1 -> N tasks`
- `task 1 -> N comments`
- `task 1 -> N events`
- `task 1 -> N external_links`

## Минимальные backend-поля для совместимости с текущим MVP

Если запускать backend итеративно, то в первой серверной версии достаточно
обязательных полей:

- `id`
- `title`
- `description`
- `status`
- `priority`
- `ownerId`
- `modelId`
- `createdAt`
- `updatedAt`

Тогда фронтенд можно будет постепенно перевести:

1. от `owner` к `ownerId` + справочник пользователей;
2. от `model` к `modelId` + каталог моделей;
3. от локального массива к HTTP API без полной переделки UI.

## Правила аудита и истории

Для каждой существенной операции стоит писать событие в `task_events`:

- создание задачи;
- изменение статуса;
- изменение приоритета;
- смена владельца;
- привязка/отвязка внешней интеграции;
- архивирование задачи.

Это позволит:

- восстановить хронологию работы;
- считать lead time и cycle time;
- строить activity feed в интерфейсе;
- поддержать безопасную синхронизацию с внешними системами.

## Миграция от текущего MVP

### Шаг 1. Нормализация данных в API

- сохранить текущие статусы и приоритеты;
- научить сервер принимать текстовые `owner` и `model`, но на своей стороне
  резолвить их в справочники;
- возвращать фронтенду уже нормализованный ответ.

### Шаг 2. Появление системных сущностей

- добавить экран управления участниками workspace;
- подключить каталог моделей;
- хранить события изменений.

### Шаг 3. Переход на совместную работу

- включить авторизацию;
- добавить optimistic updates;
- завести серверную синхронизацию комментариев и внешних ссылок.

## Открытые вопросы

- Нужен ли отдельный тип сущности `Experiment`, если одна задача порождает
  несколько сравнительных запусков моделей?
- Стоит ли хранить cost/usage метрики по модели в рамках задачи уже в первой
  backend-версии?
- Будет ли у одной задачи только один владелец или нужны соисполнители?
